# MANUGA (마뉴가) PRD v1.0

> 챗GPT 기획 대화 전체 + 피그마 디자인 초안을 종합해서 정리한 마뉴가 공식 설계서입니다. 앞으로는 이 문서를 기준으로 개발합니다.

## 1. 서비스 정의

**마뉴가(MANUGA)** = 생활패턴 기반 영양소 추천 엔진

생활환경 + 활동량 + 건강목표 + 현재고민을 입력받아 맞춤 영양소를 추천하고, 사용자가 추천받은 것/직접 고른 것을 캐비닛에서 관리하며 매일 복용 체크하는 헬스케어 플랫폼.

**아닌 것** (대화에서 두 번 헷갈렸던 부분, 명확히 배제)
- 건강 MBTI / 유형 분류 앱 (수면개선형, 운동강화형 등 ❌)
- 습관관리·챌린지 앱 (연속출석 자체가 목적 ❌)
- 의료적 진단 앱 ("OO 부족합니다" 같은 진단 표현 ❌)

## 2. 타겟 고객

사무직·좌식근무 등 생활패턴이 비교적 일정한 성인. 영양제에 관심은 있지만 "내 생활에 어떤 성분이 필요한지" 판단이 어려운 사람.

## 3. 핵심 가치 제안

"OO에 좋은 영양제"는 검색만 해도 유튜브·블로그·AI가 다 알려줘서 앱을 설치할 동기가 약함. 마뉴가의 차별점은 직접 검색할 필요 없이 내 생활환경을 입력하면 추천이 나오고, 추천 이유가 같이 제시되어 신뢰가 생기고, 추천받은 것과 직접 추가한 걸 한 곳(캐비닛)에서 관리하면서 매일 체크하는 루틴이 만들어진다는 것.

## 4. 온보딩 설문 (v1, 12문항, 확정)

설문 철학: 건강 상태를 진단하는 게 아니라 생활환경 데이터를 수집하는 것. 1분 내 완료 목표.

1. 성별 — 남성 / 여성
2. 연령대 — 20대 / 30대 / 40대 / 50대 / 60대+
3. 컴퓨터 사용시간 — 1시간 이하 / 1~4시간 / 4~8시간 / 8시간 이상
4. 하루 평균 앉아있는 시간 — 1시간 이하 / 1~3 / 3~6 / 6~8 / 8시간 이상
5. 하루 평균 걷는 시간 — 30분 이하 / 30분~1시간 / 1~2시간 / 2시간 이상
6. 하루 평균 햇빛 노출 — 거의 없음 / 30분 이하 / 30분~1시간 / 1시간 이상
7. 업무상 신체활용 강도 — 매우낮음(주로 앉아서 근무) / 낮음 / 보통(서있고 이동 있음) / 높음 / 매우높음(중량물·육체노동)
8. 운동 빈도 — 안함 / 주1~2회 / 주3~4회 / 주5회 이상
9. 건강 목표 (최대 2개) — 눈건강 / 운동회복 / 근육증가 / 면역관리 / 장건강 / 혈행건강 / 관절관리 / 활력관리
10. 현재 고민 (최대 3개) — 눈피로 / 만성피로 / 수면부족 / 자고 일어나도 피곤함 / 근육회복 느림 / 관절불편 / 장트러블 / 특별한 고민 없음
11. 식습관 — 균형있게 먹음 / 육류위주 / 탄수화물위주 / 배달음식위주 / 불규칙함
12. 수면 상태 — 매우좋음 / 보통 / 부족함 / 매우부족함

대화 중에 한 번 더 등장했던 "수면·운동·스트레스 정밀점수형" 설문은 폐기합니다. 피그마 디자인이 위 12문항 구조(생활환경+목표+고민 기반)를 따르고 있어서 이걸 정본으로 확정했습니다.

## 5. 영양소 카탈로그 (Core12) — 확정 필요

피그마에서 확인된 6개: 비타민B군, 비타민C, 비타민D, 오메가3, 루테인, 유산균
챗 히스토리 Core8 기준 추가 후보: 마그네슘, 아연, 단백질
관절 관리 카테고리 배정 과정에서 추가 확정: 칼슘
나머지 2개 확정: 비타민A(눈건강 관련 영양소), 비타민E(혈행건강 관련 영양소) — 둘 다 매핑 테이블에서 이미 "관련 영양소"로 쓰이고 있어서 카탈로그에도 포함

→ Core12 전체 확정 완료 (비오틴은 매핑 테이블 어디에도 안 쓰여서 제외)

## 6. 추천 로직 — 카테고리 ↔ 영양소 매핑 (신규 설계 초안)

피그마 분석 화면은 영양소 이름이 아니라 건강 카테고리(눈건강/피로관리/혈행건강 등)로 결과를 묶어서 보여줍니다. 이 매핑은 기존 대화에서 설계되지 않았던 부분이라 초안만 잡았습니다 — 코딩 들어가기 전에 같이 다듬어야 합니다.

- 눈 건강 → 루테인 (+ 비타민A 후보)
- 피로 관리 → 마그네슘, 비타민B군
- 혈행 건강 → 오메가3
- 면역 관리 → 비타민C, 아연, 비타민D (확정: 햇빛노출 부족 신호가 여기로 흐름)
- 장 건강 → 유산균
- 관절 관리 → 칼슘, 마그네슘 (확정)
- 활력 관리 → 비타민B군, 마그네슘
- 운동 회복 → 단백질, 마그네슘
- 근육 증가 → 단백질, 아연

점수는 내부 계산용일 뿐 사용자에게는 절대 노출하지 않습니다 (예: "마그네슘 92점" ❌). 가중치 예시:

- 컴퓨터 사용시간 8시간 이상 + 현재고민에 눈피로 선택 → 눈건강(루테인) 비중 상승
- 햇빛노출 거의없음 → 면역관리(비타민D) 비중 상승
- 운동빈도 안함 + 건강목표에 운동회복 선택 → 오메가3·단백질 비중 상승
- 만성피로·수면부족 선택 → 마그네슘·비타민B군 비중 상승

**분석 이유는 카테고리당 고정 문장이 아니라, 실제로 점수에 기여한 답변들을 동적으로 조합해서 보여줘야 합니다.** 예: 눈건강은 컴퓨터 사용시간이 길었을 때와 "눈피로"를 고민으로 선택했을 때 둘 다 점수가 올라가는데, 사용자가 둘 다 선택했다면 두 이유를 같이 보여주고 하나만 선택했다면 그 이유만 보여줘야 합니다. 이게 마뉴가의 핵심 자산("왜 추천했는지")이라 고정 문장으로 때우면 안 됩니다.

## 7. 결과 화면 규칙

- 분석 결과는 영양소 단위가 아니라 건강 카테고리 단위로 먼저 보여줌
- 카테고리 탭 → 상세화면: 분석 이유 + 관리 제안 + 함께 고려할 영양소 + "캐비닛에 담기" 버튼
- 점수(숫자) 노출 절대 금지
- "부족합니다" 진단 톤 금지 → "고려해볼 수 있습니다" 제안 톤 사용
- 특정 브랜드·제품 구매 유도 금지 (영양소 추천까지만)
- "다시 분석하기"로 재설문 가능, 최근 분석일 표시
- "결과 보기" 버튼은 TOP3 카테고리(분석 이유, 관리 제안, 관련 영양소, 캐비닛에 담기)를 한 화면에 순서대로 모아서 보여주는 리포트 화면으로 이동합니다. 카테고리 하나씩 따로 들어가지 않고 스크롤로 전체를 한 번에 훑어볼 수 있게 합니다.

## 8. 캐비닛 (건강 캐비닛)

사용자가 관리 중인 영양소 리스트. 분석 결과에서 "캐비닛에 담기"로 추가하거나 Core12 검색에서 직접 추가. 항목별 추가일·메모 기록, 삭제 가능.

## 9. 홈 (오늘의 건강 관리)

캐비닛에 담긴 항목을 매일 체크하는 화면. 진행률(예: 2/3)과 연속 관리일(스트릭) 표시. "습관관리 앱이 되지 말 것"이라는 원칙과 충돌하지 않도록, 범위는 "내가 고른 영양제 복용 체크"로 한정합니다.

## 10. 화면 흐름 / 정보구조

하단 탭: 홈 / 분석 / 캐비닛 (3탭)

흐름: 웰컴 화면(첫 진입, 앱 소개 + 시작하기) → 온보딩 설문작성 → 생활패턴 분석 결과확인 → 결과 상세확인 → 캐비닛 관리목록확인 → 홈 매일체크 (이후 반복)

**웰컴 화면 (신규):** 설문 데이터가 없는 사용자가 처음 들어왔을 때 보여주는 화면. 12문항을 맥락 없이 마주치는 이탈을 줄이기 위한 용도이며, 설문을 건너뛰는 기능은 아님(설문 데이터 없이는 추천이 불가능하므로 스킵 버튼은 넣지 않음).

긴 설명문 한 번에 보여주는 대신, 짧은 문장을 하나씩 순서대로 페이드인-페이드아웃시키는 방식(아이폰 설정 화면 같은 느낌)으로 진행:
1. "안녕하세요!"
2. "마뉴가입니다"
3. "당신의 하루를 살펴보고, 필요한 영양소를 찾아드릴게요"

마지막에는 사라지지 않고 고정되는 화면: 로고(잎 아이콘 + MANUGA, 다른 화면 헤더와 동일한 스타일, 크게) + "1분이면 충분해요" + "시작하기" 버튼.

로고는 Home.jsx 헤더와 동일한 스타일(emerald-100 원형 배지 안 Leaf 아이콘 + emerald-700 색 MANUGA 글자, 색이 둘로 안 갈라지게)로 맞추되 웰컴 화면에서는 더 크게 표시.

하단 "시작하기" 버튼 → 온보딩 설문으로 이동. 디자인은 기존 화면과 동일(stone-50 배경, 흰 카드, emerald 포인트, MANUGA 로고).

RootGuard 로직 변경: localStorage에 onboardingAnswers 없음 → /welcome (기존: /onboarding으로 직행 → 변경: /welcome을 먼저 거침)

## 11. 기술 스택

React + Vite 기반 웹앱(PWA)으로 진행합니다. 나주숲교회 앱과 동일하게 Firebase Hosting을 주 배포처로, Vercel을 GitHub push 시 자동배포 보조로 사용. 데이터 저장은 Firestore를 우선 검토(교회 앱과 스택 통일). 추후 트래픽이 생기면 Capacitor로 래핑해서 앱스토어 배포도 검토(SRPG 게임 때 세웠던 계획과 동일한 전략).

Flutter 대비 이쪽을 선택한 이유: 어제 Flutter 환경설정·Navigator 디버깅에 많은 시간이 들었던 반면, React+Firebase/Vercel 조합은 이미 익숙하고 자동화된 배포 흐름(`npm run release`)도 갖고 있어서 Claude Code로 바로 반복 작업하기 더 빠릅니다.

**네이티브 전환 경로 (앱스토어·플레이스토어 출시 목표 확정)**

웹앱이 어느 정도 완성되면 Capacitor로 그대로 감싸서 두 스토어에 낼 수 있습니다. HealthKit처럼 깊은 네이티브 API가 필요한 게 아니라서(애플워치 연동 같은 문제는 RunStats 쪽 이슈고 마뉴가는 해당 없음) React Native로 다시 짤 필요는 없고, 지금 짜는 React 코드를 거의 그대로 재사용합니다.

- iOS: Apple Developer Program 연 99달러. Xcode 빌드·서명에 Mac이 필요(없으면 Codemagic 같은 클라우드 빌드 서비스로 대체 가능). 단순 웹뷰처럼 보이면 리뷰에서 반려될 수 있어서, 지금 피그마처럼 네이티브 느낌 나는 하단 탭 UI + 오프라인 동작을 갖추는 게 통과에 유리합니다.
- Android: Google Play Console 1회 25달러. 다만 개인 계정은 정식 출시 전에 테스터 12명을 14일 연속 모집해서 비공개 테스트를 통과해야 하는 규정(2023년 시행, 2026년 현재도 유지)이 있어서 시간 여유를 두고 준비하는 게 좋습니다. 법인 계정은 이 테스트 요건은 면제되지만 D-U-N-S 번호가 필요합니다.
- 지금 단계에서 미리 챙겨둘 것: 복용 알림 기능을 만들 때 브라우저 푸시보다 Capacitor의 네이티브 로컬 알림 플러그인을 쓸 수 있게 구조를 열어두는 것, 건강 관련 데이터를 다루는 만큼 두 스토어 제출 시 개인정보처리방침 URL이 필요하다는 것.

## 12. MVP 범위

1차: 설문 12문항 → 매핑 룰셋 기반 추천 → 카테고리별 결과(TOP3 + 이유) → 캐비닛 담기/직접추가 → 홈 체크리스트. 로그인 없이 로컬 저장으로 우선 동작.

2차(이후): 로그인, 클라우드 동기화(Firestore)
3차(이후): AI 기반 추천 보강(룰셋 → LLM), 아이허브 등 최저가 연동
4차(이후): 구독 모델

## 13. 수익 모델 (장기 구상, 재검토 필요)

대화 초반엔 아이허브·쿠팡파트너스 제휴 링크가 핵심 수익 모델로 나왔는데, 이후 합의된 "특정 제품 추천 안 함" 원칙과 직접 충돌합니다. 코딩보다 먼저 방향을 정해야 할 부분 — 예를 들어 "영양소 추천까지는 무료, 제품 연결은 별도 화면에서 선택적으로 노출" 같은 절충안이 필요해 보입니다. 장기적으로는 구독 모델(월 2,900~4,900원, AI 고도화 버전)도 후보입니다.

**2차 복합 제품 DB 방향 (검토 중):**
센트룸처럼 비타민C+D+아연이 한 제품에 다 들어있는 복합 영양제를 지금 구조(영양소 단위 캐비닛)로는 자연스럽게 담기 어려워요. 2차에서 제품 단위 추가를 지원할 때 자체 DB 구축은 현실적으로 어렵고, 두 가지 방향 검토 중:
- 식품안전처 건강기능식품 공공데이터(무료, 국내 제품 성분·함량 포함) 연동
- 바코드 스캔 → Open Food Facts 오픈 DB 활용

MVP 1차에서는 복합 제품 사용자는 포함 영양소를 각각 캐비닛에 추가하고 메모에 제품명 기록하는 방식으로 안내.

## 14. 영양소 콘텐츠 자산 (Core12 설명)

각 영양소 카드에 들어갈 설명 콘텐츠입니다. 톤앤매너: 긍정적, 단정하지 않음, 음식 우선, 실천 제안, 부담 없는 표현. 의학적 효능 단정 표현 대신 "~에 필요한/도움을 줄 수 있는" 식으로 작성합니다.

| 영양소 | 한 줄 소개 | 이런 분께 | 식품 예시 |
|---|---|---|---|
| 비타민C | 정상적인 면역기능 유지와 항산화 작용에 필요한 영양소입니다. | 과일과 채소 섭취가 적거나 식사가 불규칙한 분 | 귤, 오렌지, 키위, 딸기, 파프리카 |
| 비타민D | 뼈 건강과 면역기능 유지에 필요한 영양소입니다. | 실내 활동이 많거나 햇빛을 충분히 쬐기 어려운 분 | 연어, 고등어, 달걀노른자 |
| 비타민A | 눈 건강과 피부 건강 유지에 필요한 영양소입니다. | 장시간 스마트폰이나 컴퓨터를 사용하는 분 | 당근, 단호박, 시금치 |
| 루테인 | 눈의 황반 건강 유지에 도움을 줄 수 있는 성분입니다. | 장시간 화면을 보거나 눈 피로를 자주 느끼는 분 | 케일, 시금치, 브로콜리 |
| 비타민B군 | 에너지 생성과 활력 있는 일상 유지에 필요한 영양소입니다. | 업무·학업으로 바쁜 일상을 보내는 분 | 현미, 돼지고기, 달걀 |

| 오메가3 | 혈행 건강과 혈중 중성지질 개선에 도움을 줄 수 있는 영양성분입니다. | 생선 섭취가 적은 분 | 고등어, 연어, 참치 |

| 아연 | 정상적인 면역기능 유지에 필요한 영양소입니다. | 균형 잡힌 영양 관리에 관심 있는 분 | 굴, 소고기, 견과류 |
| 유산균 | 장 건강과 원활한 배변 활동에 도움을 줄 수 있는 성분입니다. | 식습관이 불규칙하거나 장 건강에 관심 있는 분 | 요구르트, 김치, 발효식품 |

공통 안내문(모든 영양소 카드 하단 고정 문구): "균형 잡힌 식사와 건강한 생활습관이 영양 관리의 가장 좋은 출발점입니다."

**향후 결과 화면 흐름 개선안 (별도 작업, 아직 미구현):** 응원 한마디 → 관심 영역(카테고리) → 분석 이유 → 관련 영양소 → 실천 팁(식품 예시 기반) → 안내문(공통 문구) 순서로 재구성. 현재는 카테고리/이유/제안/관련영양소/캐비닛담기까지만 구현되어 있고, 응원 한마디·실천팁·안내문은 아직 화면에 없음.

## 16. 복합 영양제 묶기 기능 (2차 작업)

아이폰 홈화면에서 앱을 드래그해서 겹치면 폴더가 만들어지는 것처럼, 캐비닛에서 영양소 항목을 꾹 눌러서 다른 항목 위로 드래그해 겹치면 하나의 복합 영양제 카드로 묶이는 기능.

흐름: 꾹 누르기 → 편집 모드 진입(항목들 살짝 흔들리는 애니메이션) → 다른 항목 위로 드래그해서 겹치기 → "복합 영양제로 묶기" 확인 → 제품 이름 직접 입력(예: "센트룸 이뮨부스트") → 캐비닛에 폴더형 카드로 표시(예: 센트룸 이뮨부스트 · 비타민C, 비타민D, 아연 포함) → 홈 체크리스트에서도 개별 항목 대신 제품명 하나로 표시.

핵심 가치: 외부 DB나 API 없이 사용자가 직접 "내가 먹는 복합 영양제"를 앱이 이해하게 만드는 방식. PRD 13번의 복합 제품 DB 문제를 가장 현실적으로 해결.

구현 시 고려사항: 모바일 웹에서 드래그 앤 드롭은 터치 이벤트 처리가 복잡하므로 dnd-kit 같은 라이브러리 사용 권장. Capacitor 네이티브 앱 전환 후 구현이 더 자연스러울 수 있음.

안내 문구: 캐비닛 첫 진입 시 한 번만 표시(localStorage로 노출 여부 관리). "항목을 길게 눌러 다른 항목과 합칠 수 있어요" 툴팁 또는 배너 형태. 합치기 기능 구현과 동시에 추가.
